================================================================================
QUY ƯỚC ĐẶT TÊN FILE TÀI LIỆU (Documentation Naming Convention)
================================================================================

MỤC ĐÍCH
--------
Thống nhất cách đặt tên các file documentation trong thư mục docs/,
giúp dễ quản lý, tìm kiếm và sắp xếp theo logic.


NGUYÊN TẮC ĐẶT TÊN
------------------

Format: [số thứ tự]_[loại]_[tên-mô-tả].[txt|md]

1. SỐ THỨ TỰ (2 chữ số: 01, 02, 03... hoặc 3 chữ số nếu > 99)
   - Giúp sắp xếp file theo thứ tự ưu tiên/thời gian
   - File này (naming convention) là 00

2. LOẠI TÀI LIỆU (category)
   - feature:       Tài liệu về tính năng mới
   - architecture:  Kiến trúc, design pattern
   - bugfix:        Báo cáo sửa lỗi
   - config:        Cấu hình, setup
   - optimization:  Tối ưu hóa hiệu năng
   - guide:         Hướng dẫn sử dụng
   - migration:     Quá trình chuyển đổi/migrate
   - audit:         Đánh giá, kiểm tra

3. TÊN MÔ TẢ
   - Viết thường, không dấu
   - Dùng dấu gạch ngang (-) thay khoảng trắng
   - Ngắn gọn, mô tả rõ nội dung

4. ĐỊNH DẠNG FILE
   - .txt: Tài liệu text thuần
   - .md:  Tài liệu Markdown (có format)


VÍ DỤ
-----

01_feature_contact-guide.txt      → Feature về contact/search users
02_architecture_mvvm-migration.txt → Chuyển đổi sang kiến trúc MVVM
03_bugfix_file-management.txt     → Sửa lỗi liên quan đến quản lý file
04_config_firestore-security.txt  → Cấu hình Firestore security rules
32_migration_api-deployment.txt   → Tài liệu migration API deployment


LƯU Ý
-----

- Khi thêm file mới, chọn số thứ tự tiếp theo (hiện tại: 95+)
- Nếu có nhiều file cùng loại, sắp xếp theo thứ tự logic hoặc thời gian
- Có thể thêm loại mới nếu cần thiết
- Tất cả file documentation thuộc dự án nên đặt trong folder docs/
- File build artifacts (trong app/build/) KHÔNG cần tổ chức lại
- README.md trong các subfolder (server/, admin-web/, scripts/) giữ nguyên


================================================================================
QUY ƯỚC TRÌNH BÀY NỘI DUNG (Content Formatting Convention)
================================================================================

Để đảm bảo tính nhất quán, các file tài liệu nên tuân theo cấu trúc sau:

1. HEADER
   - Dùng 80 dấu bằng (=) để phân cách header
   - Tiêu đề viết HOA

   Ví dụ:
   ================================================================================
   TÊN TÍNH NĂNG (Feature Name)
   ================================================================================

2. METADATA
   - NGÀY TẠO: DD/MM/YYYY
   - LOẠI: Feature/Bugfix/Architecture...
   - TÍNH NĂNG: Mô tả ngắn gọn
   - KIẾN TRÚC: MVVM/Repository...

3. CẤU TRÚC MỤC (SECTIONS)
   - Dùng 80 dấu bằng (=) để phân cách các mục lớn
   - Đánh số 1. 2. 3.
   - Tiêu đề mục viết HOA

4. TIỂU MỤC (SUBSECTIONS)
   - Đánh số 1.1, 1.2...
   - Tiêu đề viết HOA

5. BẢNG BIỂU (TABLES)
   - Sử dụng ASCII art hoặc Markdown table

   ASCII:
   ┌─────────────────┬─────────┬─────────────────────────┐
   │ Field           │ Type    │ Mô tả                   │
   ├─────────────────┼─────────┼─────────────────────────┤
   │ fieldName       │ String  │ Description             │
   └─────────────────┴─────────┴─────────────────────────┘

   Markdown:
   | Field     | Type   | Mô tả       |
   |-----------|--------|-------------|
   | fieldName | String | Description |
